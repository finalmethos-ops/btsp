# 001S.2 — PO Splitting and Consolidation

## Grouping order

The generator applies boundaries in this order:

1. Workflow code — BPP and Independent never mix.
2. Vendor code.
3. Currency.
4. Source request when consolidation is disabled.
5. Store when store grouping is enabled.
6. Configured maximum lines and maximum total.

Every resulting PO receives its own atomic number. A Purchase Request may appear as a source on more
than one PO only when its lines were split during the same generation operation. Any later attempt to
generate from that source is rejected.

## Configuration

Configuration uses scope `purchase_order / default`.

| Key | Default | Behavior |
| --- | --- | --- |
| `consolidation.enabled` | `{"enabled":true}` | Combine compatible requests |
| `consolidation.by_store` | `{"enabled":false}` | Keep stores on separate POs |
| `split.max_lines` | `{"count":0}` | Maximum lines per PO; zero is unlimited |
| `split.max_total` | `{"amount":0}` | Maximum PO total; zero is unlimited |

Malformed or negative limits fail closed. A single line whose value exceeds `split.max_total` remains
whole on its own PO; line quantities are never silently divided.

## Financial integrity

Each split header is recalculated from only its included lines:

```text
subtotal = sum(quantity × unit price)
total = subtotal + freight + tax
```

The sum of all split PO totals equals the source request total. Source request ID, source line ID, and
store number remain attached to every generated line.

## Validation

Automated coverage verifies line-count splitting, total partitioning, store grouping, distinct PO
numbers, repeated source links within one split operation, and aggregate financial equality.
