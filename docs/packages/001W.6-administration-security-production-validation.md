# 001W.6 — Administration Security and BTSP v1.0 Production Validation

## Outcome

001W.6 closes Epic 001W and the BTSP v1.0 capability baseline with consolidated administration authorization, integrity, deployment, and release validation.

## Security controls

- Administrative APIs remain permission-gated; frontend navigation is convenience, not an authorization boundary.
- System roles are release-managed and cannot be changed or deleted through the runtime role API.
- Custom roles can contain only registered permissions and cannot be deleted while assigned.
- Workflow activation preserves existing instances and permits only one active version per workflow code.
- Notification templates allow only simple placeholders and allow-listed recipient configuration.
- Public health probes disclose no dependency details; protected diagnostics require `system.health.read`.
- Audit export is independently protected by `audit.export` and escapes spreadsheet formulas.
- Administrative mutations retain immutable actor-attributed snapshots.

## Integrity audit

Run the read-only production audit after migrations and release seeding:

```bash
cd backend
python -m scripts.validate_001w_integrity
```

The audit fails closed on missing or unexpected system roles, missing `SYSTEM_ADMIN` administration permissions, multiple active workflow versions, or unattributed administrative snapshots. Output contains identifiers only for invalid audit records and no credentials or business-document payloads.

## Release boundary

The v1.0 baseline provides the application capabilities in the roadmap. Production approval still requires a clean immutable revision, successful CI and migration rehearsal, dependency review, backup/restore evidence, environment-specific security review, and recorded human approvals. Automated validation does not create those approvals.

No schema migration is required beyond `0028_audit_reporting` delivered in 001W.5.
