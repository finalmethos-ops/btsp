# 001R.1 — Purchase Request Domain

## Delivered foundation

- Master purchase request and owned line-item persistence.
- References to official stores, internal catalog vendors/products, and workflow instances.
- Draft CRUD, line add/edit/delete, calculated subtotal/freight/tax/total.
- Catalog price snapshots on transactional lines.
- Submission into either `BPP_PURCHASING` or `IND_PURCHASING`.
- Workflow permission, store eligibility, and region-scope enforcement at submission.
- Immutable created, workflow-started, and submitted snapshots.

## API

- `POST /api/v1/purchase-requests`
- `GET /api/v1/purchase-requests`
- `GET|PATCH|DELETE /api/v1/purchase-requests/{id}`
- `POST /api/v1/purchase-requests/{id}/line-items`
- `PUT|DELETE /api/v1/purchase-requests/{id}/line-items/{line_id}`
- `POST /api/v1/purchase-requests/{id}/submit`

Non-administrators can access only requests they created. Submitted requests are locked against
draft mutation. Submission requires the workflow-specific submit permission.

## Validation

```sh
docker compose exec backend alembic upgrade head
docker compose exec backend pytest
docker compose exec backend ruff check app tests
```

## Follow-on scope

Draft auto-save/recovery/expiration/cloning, attachments, configurable business rules, and the
purchase editor UI are delivered in their named 001R subpackages. Workflow transitions synchronize
the `purchase_requests.status` read projection while the immutable workflow events remain the
authoritative lifecycle record.
