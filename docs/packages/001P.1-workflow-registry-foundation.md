# Implementation Package 001P.1 — Workflow Registry Foundation

## Objective

Create a canonical registry for BTSP workflow identity and routing before adding workflow-specific business packages.

## Scope

This package adds:

- Typed workflow registration and registry contracts
- Canonical BPP and Independent metadata
- Duplicate and unknown workflow-code validation
- Protected workflow registry API
- Registry-backed available-workflow API responses
- Workflow engine registration validation
- Shared backend workflow code definition
- Frontend consumption of backend-owned workflow names and routes
- Registry unit and schema tests
- Workflow registry architecture documentation

## API Surface

- `GET /api/v1/workflows/available`
- `GET /api/v1/workflows/registry`

The available-workflow response retains `code` and `route` and adds the registry-owned `name` field.

## Architecture Rules Supported

- BPP and Independent workflows remain separate.
- Workflow identity and routing have one backend source of truth.
- Runtime state-machine definitions remain versioned in the workflow engine.
- Role assignment and permissions remain authoritative for access.
- Variable workflow behavior remains configuration-backed.

## Notes

This foundation does not add BPP or Independent process definitions. Those remain separate workflow-specific packages built on the registry and engine contracts.
