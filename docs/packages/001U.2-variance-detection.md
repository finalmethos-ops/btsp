# 001U.2 — Variance Detection

## Outcome

001U.2 turns receiving discrepancies into durable, actionable records rather than transient UI warnings.

## Classifications

- `order_overage`: cumulative accepted quantity exceeds the ordered quantity.
- `rejected_quantity`: a receiver records damaged or otherwise rejected physical units.
- `asn_shortage`: physical quantity is below the selected ASN line quantity.
- `asn_overage`: physical quantity exceeds the selected ASN line quantity.

Order overages and rejected quantities are exceptions. ASN mismatches are warnings because an ASN is vendor-advisory evidence, not the physical ledger.

## Lifecycle

Every variance begins `open`. An authorized operator may `resolve` it after corrective action or `waive` it with a required explanation. Closing an already closed variance fails. A receipt remains `posted_with_exceptions` while any variance is open and returns to `posted` when all are closed.

Both receipt statuses count toward cumulative PO acceptance. Exception classification never removes accepted physical inventory from the ledger.

## API and UI

- `GET /api/v1/receipts/variances/open`
- `POST /api/v1/receipts/variances/{variance_id}/resolution`
- Receiving workspace: `/receiving`

Resolution requires `receiving.variances.manage` and follows the same store-authority boundary as receipt posting.

Tolerance policies and backorder decisions remain separate. 001U.3 will consume shortage evidence to create explicit backorder/resolution records.
