# 001W.4 — System Health and Operational Diagnostics

## Outcome

001W.4 adds protected operational diagnostics while preserving minimal, non-sensitive public liveness and readiness probes.

## Capabilities

- Report application version, environment, process uptime, and database revision.
- Measure database and Redis availability and response latency.
- Verify attachment, purchase-order export, and analytics-report storage writability and free space.
- Count failed notification deliveries, analytics report runs, and connector imports/dead letters.
- Classify the system as healthy, degraded, or unavailable.
- Refresh diagnostics from an administrator workspace.

## Probe boundary

- `GET /api/v1/health` is public and reports only process liveness.
- `GET /api/v1/ready` is public, reports no dependency details, and returns HTTP 503 when database or Redis is unavailable.
- `GET /api/v1/system/diagnostics` requires `system.health.read` and exposes operational detail.
- Administration workspace: `/admin/system-health`

Dependency error details expose exception class only, not connection strings, credentials, hostnames, or filesystem paths.

## Deployment correction

The production Compose override now retains the durable analytics-report volume. `ANALYTICS_REPORT_PATH` is also documented in the environment template.

No database migration is required.
