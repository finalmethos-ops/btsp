# 001S.4 — Internal Transmission Lifecycle

## Boundary

This package records operator-controlled Purchase Order handoff. It does not send email, call a
vendor API, produce EDI, invoke a webhook, or claim external delivery automatically.

Supported channels are labels for internal operations:

- `manual`
- `secure_file`
- `internal_email`

## Eligibility

A transmission requires:

- An immutable PDF, CSV, or JSON artifact belonging to the PO.
- Every source Purchase Request at workflow state `vendor_submission`.
- `workflow.bpp.vendor_submit` for BPP, or the existing `workflow.ind.review` permission for
  Independent purchasing.
- No existing transmission for the same artifact.

## State machine

```text
prepared --release--> ready --mark_delivered--> delivered
                            --mark_failed-----> failed --retry--> prepared
prepared / ready / failed --cancel----------> cancelled
```

Failure requires a reason. Invalid actions are rejected. Every transition appends a row to
`purchase_order_transmission_events`; rows are never updated or deleted through the API. The PO status
is projected as `transmission_prepared`, `transmission_ready`, `transmission_failed`,
`transmission_cancelled`, or `transmitted`.

## API

- `POST /api/v1/purchase-orders/{id}/transmissions`
- `GET /api/v1/purchase-orders/{id}/transmissions`
- `POST /api/v1/purchase-orders/{id}/transmissions/{transmission_id}/actions`

Creation and every status change also produce immutable platform snapshots. Destination and notes
are audit labels only; they do not activate an external connector.

## Validation

Automated coverage verifies early-transmission rejection, missing permission, artifact ownership,
release, failure reason, retry, delivery, invalid terminal actions, PO status projection, event
history, and snapshots.
