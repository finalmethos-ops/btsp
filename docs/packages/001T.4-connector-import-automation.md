# 001T.4 — Connector Import Automation

## Outcome

001T.4 adds a bounded, replay-safe import boundary between vendor transports and BTSP's existing normalized inbound-event model.

## Capabilities

- Import one normalized JSON event or a batch of up to 1,000 events.
- Accept imports only for active inbound or bidirectional endpoints.
- Limit uploaded content to 10 MiB before parsing.
- Record the source filename, media type, SHA-256 checksum, actor, result, event count, and error.
- Link newly created inbound events to their connector import run.
- Return the existing run when the same endpoint receives identical file bytes again.
- Report partial failure honestly when an external event ID conflicts after earlier events were accepted.

## HTTP API

- `POST /api/v1/vendor-integrations/endpoints/{endpoint_id}/imports`
- `GET /api/v1/vendor-integrations/imports?endpoint_id={endpoint_id}`

Imports require `vendor.integrations.ingest`; import-run visibility requires `vendor.integrations.read`.

## Adapter boundary

REST API, SFTP, and manual-import endpoints accept normalized JSON through the same adapter contract. The endpoint ID is supplied by the authenticated route and cannot be overridden by file content.

EDI endpoints fail closed. BTSP will not infer an X12 or EDIFACT transaction mapping: a future adapter must explicitly identify the standard, version, transaction set, partner profile, validation rules, and normalized field mapping.

## Deferred to 001T.5

- SFTP polling and remote file movement
- API polling and webhooks
- job scheduling, retries, and dead-letter handling
- partner-specific EDI translation
- automatic projection of imported events into acknowledgements, shipments, and ASNs
