# 001R.6 — Purchase Request Attachments

## Storage model

File bytes are stored in the persistent Docker volume mounted at `/data/attachments`. PostgreSQL
stores attachment identity, request ownership, category, original filename, generated storage name,
media type, size, SHA-256 checksum, actor, timestamps, and deletion state.

Files are never stored under a client-provided path. Generated UUID names, resolved-path checks,
size limits, media-type allow-listing, and file-signature checks protect the storage boundary.

Supported categories are `quote`, `vendor_document`, `image`, `pdf`, and
`supporting_document`. Supported formats are PDF, PNG, JPEG, XLSX, and DOCX.

## API

- `POST /api/v1/purchase-requests/{id}/attachments` — multipart `category` and `file`.
- `GET /api/v1/purchase-requests/{id}/attachments`.
- `GET /api/v1/purchase-requests/{id}/attachments/{attachment_id}/content`.
- `DELETE /api/v1/purchase-requests/{id}/attachments/{attachment_id}`.

Request ownership and system-administrator access rules are reused for every operation. Drafts and
active submitted requests accept uploads. Only draft attachments may be deleted. Each upload and
deletion writes an immutable snapshot containing the attachment identity and checksum.

## Environment

| Variable | Default | Purpose |
| --- | --- | --- |
| `ATTACHMENT_STORAGE_PATH` | `/data/attachments` | Managed storage root |
| `ATTACHMENT_MAX_BYTES` | `20971520` | Per-file size limit (20 MiB) |

The database and attachment volume must be backed up together. A restored database without its
matching volume will retain metadata but cannot serve file content.

## Known limitation

Signature validation is not malware scanning. Production operators should scan uploads at the host
or storage layer until a dedicated asynchronous malware-scanning service is introduced.
