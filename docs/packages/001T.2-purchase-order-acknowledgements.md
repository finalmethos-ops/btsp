# 001T.2 — Purchase Order Acknowledgements

## Delivered

- PO-linked accepted, accepted-with-changes, and rejected acknowledgement records.
- Vendor reference, acknowledgement time, expected ship date, reason, and structured change data.
- Strict endpoint-vendor-to-PO-vendor matching.
- Internal-transmission eligibility enforcement.
- Inbound event processing and rejection status projection.
- PO status projection to `vendor_acknowledged`, `vendor_acknowledged_changes`, or
  `vendor_rejected`.
- Event-level idempotent processing with row locks for concurrent safety.
- Immutable successful and rejected acknowledgement snapshots.
- System-administrator processing and read permissions.

## API

- `POST /api/v1/vendor-integrations/events/{event_id}/process-acknowledgement`
- `GET /api/v1/vendor-integrations/acknowledgements`
- `GET /api/v1/vendor-integrations/acknowledgements/{id}`

Rejected outcomes require a reason. Accepted-with-changes outcomes require structured change
details. Invalid, cross-vendor, wrong-event-type, and pre-transmission envelopes are marked rejected
and cannot be silently reprocessed.

## Boundary

This package records vendor responses to a PO. It does not schedule shipments, create ASNs, modify
PO lines, or automatically accept vendor-proposed changes. Changed and rejected outcomes remain
explicit exception states for later operational workflows.
