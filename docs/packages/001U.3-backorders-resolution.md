# 001U.3 — Backorders and Resolution

## Outcome

001U.3 converts eligible shortage evidence into explicit, auditable operational commitments without modifying the original purchase order or physical receipt.

## Decision boundary

A shortage variance is evidence; a backorder is an operator decision. Backorders may be created from open `asn_shortage` or `rejected_quantity` variances. Creation is idempotent per source variance and resolves that variance with the recorded decision note.

## Lifecycle

```text
open -> partially_fulfilled -> fulfilled
  |             |
  +-------------+-> cancelled
  +-------------+-> substituted
```

- Receive actions require a positive quantity and cannot exceed outstanding quantity.
- Cancellation closes the remaining commitment with a required note.
- Substitution requires an explicit alternate product code and note.
- Terminal records cannot be changed again.
- Every transition retains actor, quantity, note, prior status, and resulting status.

## API and UI

- `POST /api/v1/receipts/backorders/create`
- `GET /api/v1/receipts/backorders/list`
- `POST /api/v1/receipts/backorders/{id}/actions`
- Receiving workspace: `/receiving`

Management requires `receiving.backorders.manage`; visibility uses `receiving.read`. All operations retain the receipt store-authority boundary.
