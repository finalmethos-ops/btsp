# 001W.3 — Notification Administration

## Outcome

001W.3 adds a governed workspace for notification templates and delivery history, with audited edits and controlled failed-delivery requeueing.

## Capabilities

- List, create, edit, enable, and disable notification templates.
- Configure workflow/event binding, channel, recipient strategy, and recipient settings.
- Inspect the latest 100 delivery-ledger events with status, entity, recipients, and errors.
- Requeue rendered failed deliveries while preserving the original event identity and actor.
- Reject requeue attempts for non-failed, unrendered, or recipientless events.
- Audit template creation, template updates, and administrator requeue actions.

## Security and validation

- Template placeholders are restricted to simple context names.
- Attribute traversal, indexing, conversions, and format specifications are rejected.
- Template and identifier lengths are bounded.
- Recipient configuration is allow-listed by strategy and list sizes are bounded.
- Event history limits are constrained to 1–500.

## API and UI

- `GET/POST /api/v1/notifications/templates`
- `PATCH /api/v1/notifications/templates/{template_code}`
- `GET /api/v1/notifications/events`
- `POST /api/v1/notifications/events/{notification_id}/retry`
- Administration workspace: `/admin/notifications`
- Read permission: `notifications.read`
- Mutation permission: `notifications.manage`

This package governs queued delivery records; external email and webhook transport workers remain provider-specific integration concerns.
