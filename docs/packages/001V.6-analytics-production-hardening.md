# 001V.6 — Analytics Security and Production Hardening

## Outcome

001V.6 closes Epic 001V with authorization, input, export, artifact-integrity, query-bound, and production-runtime validation.

## Security controls

- Every analytics read and export requires `analytics.read`.
- Schedule creation and due-run execution require `analytics.reports.manage`.
- Scheduled-report parameters are allow-listed by report type and length-bounded.
- CSV text that could execute as a spreadsheet formula is escaped before export.
- Artifact downloads remain confined to the configured report root.
- Artifact size and SHA-256 are verified before every download; altered files fail closed.
- Reversed or empty date windows are rejected with HTTP 422.

## Performance controls

- Schedule and run history default to 100 records and accept a bounded limit of 1–500.
- Due schedules use indexed enabled/next-run fields and row locks with `SKIP LOCKED`.
- Artifact verification streams in one-megabyte chunks rather than loading a file into memory.
- Analytics aggregation remains database-side where practical and preserves per-currency totals.

The current analytics implementation is appropriate for operational datasets. Large historical warehouses should introduce controlled snapshots or materialized views instead of moving transaction ownership out of the domain ledgers.

## Production operations

- Scheduled reports use the durable `/data/analytics-reports` volume.
- Completed runs retain byte size, digest, status, timestamps, and actor attribution.
- Failed rendering is recorded on the run without publishing a completed artifact.
- Multiple workers cannot claim the same due schedule concurrently.

No schema migration is required beyond `0027_analytics_reports` delivered in 001V.5.
