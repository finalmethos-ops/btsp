# 001S.3 — Purchase Order PDF and Structured Exports

## Formats

- PDF: paginated human-readable Purchase Order with vendor, lines, and totals.
- CSV: one row per PO line for spreadsheet/import workflows.
- JSON: canonical structured header, source, line, and financial data.

Exports are immutable versioned artifacts. Version 1 generation is idempotent: asking for an
existing format returns the original metadata and file rather than rewriting it. Every artifact stores
its content type, byte size, SHA-256 checksum, actor, and creation time in PostgreSQL while the bytes
reside in the persistent `purchase_order_export_data` volume.

PDF rendering uses invariant metadata for stable output. JSON keys are sorted. CSV text fields that
begin with spreadsheet formula characters are escaped to prevent formula injection.

## API

- `POST /api/v1/purchase-orders/{id}/artifacts/pdf`
- `POST /api/v1/purchase-orders/{id}/artifacts/csv`
- `POST /api/v1/purchase-orders/{id}/artifacts/json`
- `GET /api/v1/purchase-orders/{id}/artifacts`
- `GET /api/v1/purchase-orders/{id}/artifacts/{artifact_id}/content`

All endpoints reuse Purchase Order workflow-scope access controls. Successful creation writes an
immutable `purchase_order.artifact_created` snapshot containing format, version, size, and checksum.

## Environment and operations

`PURCHASE_ORDER_EXPORT_PATH` defaults to `/data/purchase-order-exports`. PostgreSQL and the export
volume must be backed up at the same maintenance boundary. If metadata exists but file content is
missing, the API returns a conflict/gone response instead of silently producing different bytes for
the same artifact version.

## Validation

Automated tests verify all three formats, PDF signature, JSON contract, CSV columns, checksums,
idempotency, artifact counts, and snapshot counts.
