# 001T.5 — Connector Operations

## Outcome

001T.5 adds a durable operations control plane for automated vendor connector work. Scheduling state and execution state live in PostgreSQL, allowing the API, scheduler trigger, and connector workers to run as separate processes without losing work during restarts.

## Capabilities

- Interval schedules with pause and resume controls.
- Idempotent due-work creation per schedule and scheduled time.
- Worker claims protected by exclusive, expiring leases.
- Bounded attempts and exponential retry delay.
- Automatic dead-letter transition after attempts are exhausted or a final lease expires.
- Operator-controlled dead-letter replay.
- Optional execution linkage to a completed connector import run.
- Audit snapshots for schedule creation, execution results, and replay decisions.
- Operations UI at `/vendor-connectors`.

## Deployment boundary

BTSP does not run an in-process timer inside the web server. An operator-owned scheduler calls `POST /api/v1/vendor-integrations/connector-executions/enqueue-due`. Separately deployed workers claim jobs, resolve the endpoint's opaque `connection_reference` through an approved secret provider, perform transport work, import normalized content through 001T.4, and report the result.

This separation prevents duplicate timers when the API scales horizontally and keeps connector credentials outside the BTSP database and job payloads.

## Retry lifecycle

```text
queued -> running -> completed
             |
             +-> retry -> running
                           |
                           +-> dead_letter -> queued (operator replay)
```

Attempts use `base_retry_seconds * 2^(attempt - 1)`. The schedule snapshots its retry policy into each execution so later schedule changes do not silently alter in-flight work.

## Permissions

- `vendor.integrations.read`: view schedules and execution history.
- `vendor.connectors.operate`: create/pause schedules, enqueue due work, and replay dead letters.
- `vendor.connectors.work`: claim executions and report worker results.

Worker identities should receive only `vendor.connectors.work`; human operators should not normally receive it.
